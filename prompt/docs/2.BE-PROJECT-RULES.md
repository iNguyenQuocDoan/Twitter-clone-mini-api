# Backend Project Rules

## Tech Stack

| Technology | Version | Role |
|------------|---------|------|
| Language | TypeScript 5.x (strict mode) | Application language |
| Framework | NestJS v11 | HTTP framework + DI container |
| ORM | TypeORM | Database access layer |
| Database | MySQL 8.x | Primary data store |
| Test runner | Jest | Unit and integration tests |
| Validation | class-validator + class-transformer | DTO validation |

## 1. Feature Structure

### Top-Level Layout

```
src/
├── main.ts                         # Bootstrap: global pipes, filters, interceptors
├── app.module.ts                   # Root module — imports all feature modules
│
├── config/                         # Typed environment config — no raw process.env outside here
│   ├── app.config.ts
│   ├── database.config.ts
│   └── jwt.config.ts
│
├── core/                           # Infrastructure plumbing — no business logic
│   ├── database/
│   │   └── database.module.ts      # TypeORM forRootAsync setup
│   └── logger/
│       └── logger.module.ts
│
├── shared/                         # Cross-feature reusable code
│   ├── decorators/                 # @CurrentUser(), @Roles(), @Public()
│   ├── filters/                    # HttpExceptionFilter (global)
│   ├── guards/                     # JwtAuthGuard, RolesGuard
│   ├── interceptors/               # TransformInterceptor, LoggingInterceptor
│   ├── pipes/                      # Global ValidationPipe
│   ├── utils/                      # pagination.util.ts, hash.util.ts
│   └── types/                      # ApiResponse, PaginatedResponse, JwtPayload
│
└── features/
    ├── auth/                       # Owns: roles, users, refresh_tokens, JWT
    ├── user-profile/               # Owns: addresses
    ├── product/                    # Owns: categories, products, variants, images
    ├── cart/                       # Owns: carts, cart_items
    ├── order/                      # Owns: orders, order_items, checkout logic
    └── review/                     # Owns: reviews
```

### Feature Internal Structure

```
features/[feature-name]/
├── [feature].module.ts             # NestJS module — declares providers, imports, exports
├── [feature].controller.ts         # Route handlers, DTO binding, response shaping
├── [feature].service.ts            # Business logic and orchestration
├── repositories/
│   └── [entity].repository.ts     # TypeORM queries only — no business logic here
├── dto/
│   ├── create-[entity].dto.ts
│   ├── update-[entity].dto.ts
│   └── [entity]-response.dto.ts
├── entities/
│   └── [entity].entity.ts         # TypeORM @Entity definitions
├── types/
│   └── [feature].types.ts         # Feature-local interfaces and types
├── tests/
│   ├── [feature].service.spec.ts  # Unit tests — mock repository
│   └── [feature].controller.spec.ts
└── CONTEXT.md                      # AI assistant context file for this feature
```

> Complex features with multiple sub-resources use a `controllers/` and `services/` subfolder instead of flat files. See the product feature anatomy in `3.BE-ARCHITECTURE.md`.

## 2. Naming Conventions

| Element | Convention | Example |
|---------|------------|---------|
| Feature folders | kebab-case | `user-profile/`, `product/` |
| Files | kebab-case + layer suffix | `create-user.dto.ts`, `user.entity.ts`, `user.service.ts` |
| Classes | PascalCase | `UserService`, `CreateUserDto`, `ProductVariant` |
| Functions / Methods | camelCase | `findById()`, `createOrder()`, `hashToken()` |
| Variables | camelCase | `userId`, `cartItems`, `tokenHash` |
| Constants | UPPER_SNAKE_CASE | `MAX_CART_ITEMS`, `JWT_EXPIRES_IN` |
| Interfaces | PascalCase + `I` prefix | `IUserPayload`, `IJwtStrategy` |
| Type aliases | PascalCase + `Type` suffix | `OrderStatusType`, `PaymentMethodType` |
| Entities | PascalCase, singular | `User`, `ProductVariant`, `RefreshToken` |
| DTOs | PascalCase + `Dto` suffix | `CreateOrderDto`, `LoginResponseDto` |
| Enums | PascalCase + `Enum` suffix | `OrderStatusEnum`, `RoleEnum` |

**DO vs DON'T — naming:**

| ❌ DON'T | ✅ DO |
|----------|-------|
| `getuser.ts` | `user.service.ts` |
| `createuser()` | `createUser()` |
| `USERSERVICE` | `UserService` |
| `userPayload` (interface) | `IUserPayload` |
| `orderStatus` (type alias) | `OrderStatusType` |

## 3. Feature Isolation Rules

### Ownership

Each feature owns its entities exclusively. No other feature accesses another feature's repository or entity directly.

| Feature | Owns |
|---------|------|
| auth | `roles`, `users`, `refresh_tokens`, JWT issuance |
| user-profile | `addresses` |
| product | `categories`, `products`, `product_variants`, `product_images` |
| cart | `carts`, `cart_items` |
| order | `orders`, `order_items`, checkout transaction |
| review | `reviews` |

### Cross-Feature Communication — Allowed Patterns

```typescript
// ✅ PATTERN 1: NestJS module imports (synchronous, DI-based)
// cart.module.ts — import ProductModule to access ProductVariantRepository
@Module({
  imports: [ProductModule],   // ProductModule must export the required provider
  providers: [CartService],
})
export class CartModule {}

// ✅ PATTERN 2: EventEmitter2 for async side effects
// order.service.ts — after order confirmed
this.eventEmitter.emit('order.confirmed', {
  orderId: order.id,
  userId: order.userId,
});

// notification.listener.ts — in notification feature
@OnEvent('order.confirmed')
async handleOrderConfirmed(payload: { orderId: number; userId: number }) {
  await this.notificationService.sendOrderConfirmation(payload);
}

// ❌ FORBIDDEN: Direct internal imports across feature boundaries
import { ProductService } from '../product/product.service';    // WRONG
import { ProductEntity } from '../product/entities/product.entity'; // WRONG
```

### When to Use Each Pattern

| Pattern | Use when |
|---------|----------|
| Module import | Feature B needs to call a synchronous method on Feature A's service |
| EventEmitter | Side effect is decoupled (notifications, audit logs, analytics) |
| Shared service | Logic is needed by 3+ features — extract to `shared/` |

## 4. Mandatory Code Patterns

### Error Handling

```typescript
// ✅ Use NestJS built-in HTTP exceptions for standard cases
throw new NotFoundException(`Product #${id} not found`);
throw new BadRequestException('Insufficient stock');
throw new ConflictException('Email already registered');
throw new ForbiddenException('Cannot cancel a shipped order');

// ✅ Custom exceptions for domain-specific errors
export class InsufficientStockException extends HttpException {
  constructor(sku: string, available: number, requested: number) {
    super(
      {
        code: 'PROD_003',
        message: `Insufficient stock for SKU: ${sku}`,
        details: { available, requested },
      },
      HttpStatus.BAD_REQUEST,
    );
  }
}

// ❌ DON'T swallow errors
try {
  await this.doSomething();
} catch (err) {
  // silent catch — NEVER do this
}
```

### Validation (DTOs)

All input is validated at the controller boundary using `class-validator` before the service is called.

```typescript
// ✅ DTO with full validation
export class CreateProductDto {
  @ApiProperty({ example: 'Áo thun nam' })
  @IsString()
  @IsNotEmpty()
  @MaxLength(255)
  name: string;

  @IsNumber()
  @IsPositive()
  @Type(() => Number)
  price: number;

  @IsOptional()
  @IsString()
  description?: string;
}

// ❌ DON'T validate in service
async createProduct(name: string, price: number) {
  if (!name) throw new Error('Name required'); // validation belongs in DTO
}
```

Global pipe in `main.ts`:

```typescript
app.useGlobalPipes(
  new ValidationPipe({
    transform: true,          // auto-transform types (string → number)
    whitelist: true,          // strip unknown properties
    forbidNonWhitelisted: true, // reject requests with extra fields
  }),
);
```

### Response Format

All responses are wrapped by `TransformInterceptor`. Controllers return raw data; the interceptor applies the envelope.

```typescript
// Success — single resource
{ "success": true, "data": { "id": 1, "name": "Product" }, "message": "Created successfully" }

// Success — paginated list
{
  "success": true,
  "data": [...],
  "meta": { "page": 1, "limit": 10, "total": 100, "totalPages": 10 }
}

// Error (from HttpExceptionFilter)
{
  "success": false,
  "error": { "code": "PROD_001", "message": "Product not found", "details": {} }
}
```

### Repository Pattern

```typescript
// ✅ Complex queries belong in the repository
@Injectable()
export class ProductRepository {
  constructor(
    @InjectRepository(Product)
    private readonly repo: Repository<Product>,
  ) {}

  async findActiveWithVariants(id: number): Promise<Product | null> {
    return this.repo
      .createQueryBuilder('product')
      .leftJoinAndSelect('product.variants', 'variant')
      .where('product.id = :id', { id })
      .andWhere('product.is_active = :active', { active: true })
      .getOne();
  }
}

// ❌ DON'T put queries in service
// product.service.ts
const product = await this.productRepo.createQueryBuilder()... // WRONG — move to repository
```

### Logging

Use NestJS `Logger` at the service layer. Always include structured context (user ID, resource ID, operation).

```typescript
private readonly logger = new Logger(OrderService.name);

async checkout(userId: number, dto: CreateOrderDto): Promise<Order> {
  this.logger.log(`Checkout initiated`, { userId, cartId: dto.cartId });

  try {
    const order = await this.processCheckout(userId, dto);
    this.logger.log(`Order created`, { orderId: order.id, userId });
    return order;
  } catch (err) {
    this.logger.error(`Checkout failed`, { userId, error: err.message });
    throw err;
  }
}
```

### Database Transactions

Use `QueryRunner` explicitly for multi-step write operations. Never let partial writes succeed.

```typescript
const queryRunner = this.dataSource.createQueryRunner();
await queryRunner.connect();
await queryRunner.startTransaction();

try {
  // 1. Deduct stock
  // 2. Create order + order_items
  // 3. Clear cart
  await queryRunner.commitTransaction();
} catch (err) {
  await queryRunner.rollbackTransaction();
  throw err;
} finally {
  await queryRunner.release();
}
```

## 5. Anti-Patterns (MUST NOT)

| ❌ DON'T | ✅ DO | Why |
|----------|-------|-----|
| Import from another feature's internal path | Use NestJS module imports or events | Enforces feature boundaries |
| Put business logic in controllers | Keep orchestration in services | Controllers are routing + binding only |
| Put data queries in services | Use repository pattern | Separates concerns, enables testing |
| Use `process.env` directly in feature code | Use `ConfigService` | Centralized config, type-safe |
| Store plain passwords | Hash with bcrypt (cost factor ≥ 12) | Security |
| Link cart/order items to `products` | Link to `product_variants` | Products don't have stock — variants do |
| Use FK to `addresses` in orders | Snapshot address to JSON | Address immutability for order history |
| Create circular module dependencies | Use EventEmitter for decoupling | NestJS will fail to bootstrap |
| Use `any` type | Use explicit types or generics | Defeats TypeScript strict mode |
| Silent catch blocks | Always rethrow or log + rethrow | Hidden failures in production |

```

```

### PR Requirements

| Requirement | Detail |
|-------------|--------|
| Linked issue | PR description must reference ticket ID |
| Tests pass | All unit + integration tests in CI must be green |
| Type-check | `tsc --noEmit` must pass with zero errors |
| Lint | `eslint` must pass with zero errors |
| Reviewers | Minimum 1 team member approval |
| Scope | One feature or fix per PR — no mixed concerns |
| CONTEXT.md | Update feature `CONTEXT.md` if public API or logic changes |

## 7. Testing

### Coverage Targets

| Layer | Target | Location |
|-------|--------|----------|
| Services | ≥ 80% | `features/[f]/tests/[f].service.spec.ts` |
| Controllers | ≥ 70% | `features/[f]/tests/[f].controller.spec.ts` |
| Repositories | ≥ 60% | `features/[f]/tests/[entity].repository.spec.ts` |

### Test Structure (AAA Pattern)

```typescript
describe('CartService', () => {
  let service: CartService;
  let cartRepository: jest.Mocked<CartRepository>;

  beforeEach(async () => {
    const module = await Test.createTestingModule({
      providers: [
        CartService,
        { provide: CartRepository, useValue: { findByUser: jest.fn() } },
      ],
    }).compile();

    service = module.get(CartService);
    cartRepository = module.get(CartRepository);
  });

  describe('addItem', () => {
    it('should add new item when variant not in cart', async () => {
      // Arrange
      cartRepository.findByUser.mockResolvedValue(mockEmptyCart);

      // Act
      const result = await service.addItem(userId, { productVariantId: 1, quantity: 2 });

      // Assert
      expect(result.items).toHaveLength(1);
      expect(result.items[0].quantity).toBe(2);
    });

    it('should increment quantity when variant already in cart', async () => {
      // Arrange — cart already has variantId: 1 with qty 1
      cartRepository.findByUser.mockResolvedValue(mockCartWithItem);

      // Act
      const result = await service.addItem(userId, { productVariantId: 1, quantity: 2 });

      // Assert
      expect(result.items[0].quantity).toBe(3); // 1 + 2
    });
  });
});
```

## 8. NestJS Decorators Quick Reference

```typescript
// --- Custom decorators (shared/decorators/) ---
@CurrentUser()          // Extracts user from JWT-populated request
@Roles('admin')         // Attaches role metadata for RolesGuard
@Public()               // Bypasses JwtAuthGuard on public endpoints

// --- Guards ---
@UseGuards(JwtAuthGuard)              // Validates JWT, populates request.user
@UseGuards(JwtAuthGuard, RolesGuard)  // JWT + role check in sequence

// --- Rate limiting (auth endpoints) ---
@Throttle({ default: { limit: 5, ttl: 60_000 } })
@Post('login')
async login(@Body() dto: LoginDto) {}

// --- Swagger documentation ---
@ApiTags('orders')
@ApiOperation({ summary: 'Create order from cart (checkout)' })
@ApiResponse({ status: 201, type: OrderResponseDto })
@ApiBearerAuth()

// --- Transactions (checkout, order confirmation) ---
const queryRunner = this.dataSource.createQueryRunner();
await queryRunner.connect();
await queryRunner.startTransaction();
try {
  await queryRunner.commitTransaction();
} catch {
  await queryRunner.rollbackTransaction();
} finally {
  await queryRunner.release();
}
```
